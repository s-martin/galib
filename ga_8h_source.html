24 -------------------------------------------------------------------------------
25 Overview of who does what in the GAlibrary -- see http://lancet.mit.edu/ga/
26 -------------------------------------------------------------------------------
27 Here are the library capabilities and which parts of the library are involved:
28 
29 Genetic Algorithm
30  This object contains the operators and data needed to perform the
31  optimization. There are a few basic GA implementations in the library,
32  including the basic 'simple' genetic algorithm described by Goldberg, the
33  'steady-state' genetic algorithm, the 'incremental' genetic algorithm based
34  on the Genitor model, and the 'parallel' genetic algorithm that uses multiple
35  populations in parallel on a single CPU.
36  The base GA object defines many of the parameters and statistics useful
37  when running an experiment. It also defines the basic interface for most
38  genetic algorithms.
39 
40 Genome
41  The genome object contains the actual problem-specific data as well as the
42  mapping from data-domain to problem-domain (by way of the objective functions
43  All of the genomes are derived from a base genome class. The base genome
44  class defines the interface used by genetic algorithms to modify the data.
45  Any genome used in an experiment must be derived from the base genom class
46  as well as a user-defined data type (or one of the sample types provided in
47  GAlib). The library includes array, tree, list, and binary-string data types
48  to use if you do not want to define your own.
49 
50 Population
51  Each population contains a bunch of genomes. Populations know how to
52  select genomes. They also have housekeeping routines such as replace,
53  remove, and add. A population can be given some intelligence by using its
54  evaluation member rather than evaluting each individual on its own.
55 
56 Selection
57  Selection is implemented as a member function of the population object.
58  Any GA simply calls the population's 'select' member to get the genome that
59  it needs to work with.
60 
61 Crossover/Mating
62  Crossover is implemented as a separate object. We have various crossover
63  objects, each corresponding to a genome type. These could be templatized,
64  but that wouldn't buy us much since the behaviour depends upon the data type
65  anyway.
66  Crossover is not a member function of the genomes. I fretted over how to
67  do this properly and decided to keep crossover as an operator separate from
68  the genomes. The other viable option was to make crossover a member function
69  of the genomes, but then changing the crossover during the course of the
70  evolution becomes more difficult. Also, it is easier to define asexual (or
71  other non-standard mating schemes) with an operator that is not a member of
72  the genome object.
73  When you create a GA, you specify a genome (or population of genomes). The
74  GA asks the genome how it mates, and the genome returns a pointer to its
75  preferred crossover method. You can set the crossover explicitly on the GA
76  if you prefer. Using this scheme we can assign default crossover operators
77  so that you don't have to specify a crossover unless you want to.
78  Each genome type must have a corresponding crossover operator defined that
79  can operate on that data type. The genetic algorithms don't care about the
80  specific crossover type - they call the mating methods based upon the high
81  level interface definition.
82 
83 Mutation
84  Mutation is defined at the genome level and is implemented as a member
85  function of the base genome class. Each genome must define its own
86  mutation method that will operate on its specific data type.
87 
88 Replacement strategies
89  The replacement strategy defines how a child is inserted into a population.
90  Replacement strategies include child-replaces-parent, child-replaces-worst,
91  and child-replaces-random. These make sense only for GAs with overlapping
92  populations.
93 
94 Convergence and completion
95  The GA defines a 'done' member that calls a user-defineable completion
96  function to determine whether or not the GA is finished. Built-in completion
97  routines include population-converged, best-converged and
98  number-of-generations.
99 
100 Fitness vs Objective
101  Note the difference between fitness function and objective
102  function. In this library, the objective function is user-defined and tells
103  (on an arbitrary scale) how well a genome performs relative to other
104  genomes. The fitness function, on the other hand, takes objective function
105  scores and processes them to produce a number for each genome that
106  represents its fitness for mating/selection.
107  The selection strategies include roulette wheel, tournament, linear
108  ranking, and stochastic sampling. See the selector headers for more details
109  about each strategy and its implementation.
110  Speciation is included as a type of selection. To use the speciating
111  selector you must define a distance function. See the selector header for
112  more details.
113  Scaling methods are loosely coupled with the selection strategies. They
114  include linear scaling, sigma truncation, and ranking. See the fitness
115  headers for more details about how you can mix and match scaling methods with
116  selection strategies.
117 
118 Elitism
119  Elitism is optional. If you want the best of each generation to be carried
120  over to the next generation, turn on the elitism. If not, turn it off. This
121  only works with non-overlapping populations.
122 
123 Initialization
124  Each genome has its own initialization operator. When a GA is
125  initialized, it initializes its populations, which in turn tells each
126  chromomosome to initialize itself. You can use the initialization method to
127  bias an initial population. This can be done either at the population level
128  by customizing the population's initialization operator or at the genome
129  level by customizing the genome's initilizer.
130  Default initializers are included for most genomes. See the documentation
131  and genome headers for more details.
132 
133 -------------------------------------------------------------------------------
134 Basic usage
135 
136  You create a GA by first instantiating the non-default parts that you want to
137 use. For example, if you want a binary string genome with uniform crossover
138 and roulette wheel selection, you would define an objective function,
139 instantiate a genome, instantiate a uniform crossover operator, then
140 instantiate the GA. Since the roulette wheel selector is the default, you do
141 not need to instantiate one to use that method.
142  In every case you MUST define an objective function and you MUST instantiate
143 one copy of the genome type you want to use.
